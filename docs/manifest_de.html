Technical Manifest

Das Token-Kosten Problem
in der KI-gestützten Entwicklung

TRACE – Token-aware Realtime AI Context Engine · 2026

01 — Das Problem

Was die meisten Entwickler nicht wissen

KI-gestützte Entwicklung mit Claude, GPT oder anderen Modellen verursacht Kosten die auf den ersten Blick einfach wirken: Ich schreibe eine Nachricht, das Modell antwortet. Input-Token plus Output-Token gleich Kosten.

Das ist falsch. Und dieser Irrtum kostet – buchstäblich.

Wichtiger Punkt den viele unterschätzen: Bei jedem neuen Turn in einem Chat wird der gesamte bisherige Gesprächsverlauf als Input an das Modell übergeben – nicht nur die neue Nachricht.

Das bedeutet konkret:

# Was tatsächlich pro Turn an die API gesendet wird:

Deine neue Nachricht     = Input-Token
ALLE vorherigen Nachrichten = auch Input-Token  ← das übersehen die meisten
Systemkontext + Docs    = auch Input-Token
Antwort des Modells      = Output-Token

# Das passiert weil LLMs zustandslos sind –
# sie haben kein Gedächtnis zwischen Anfragen.
# Der Kontext muss bei jedem Call komplett neu mitgeschickt werden.
02 — Die Mathematik

Kosten steigen nicht linear – sondern progressiv

Ein Gespräch das nach 10 Turns insgesamt 50.000 Token angesammelt hat, kostet bei Turn 10 fast 100× mehr als bei Turn 1 – obwohl die eigentliche neue Nachricht gleich lang ist.

Turn Neue Nachricht Mitgeschickter Kontext Gesamt-Input Kosten-Index
1 100 Token 0 Token 100 Token 1×
5 100 Token ~4.000 Token ~4.100 Token 41×
10 100 Token ~9.800 Token ~9.900 Token 99×
50 100 Token ~49.800 Token ~49.900 Token 499×

Die Formel ist einfach: Kosten(Turn N) ≈ Kosten(Turn 1) × N

Praxisbeispiel: Eine Claude Sonnet Session mit 50 Turns und durchschnittlich 1.000 Token pro Nachricht kostet nicht 50 × $0.003 = $0.15 – sondern durch den akkumulierten Kontext ein Vielfaches davon. Der letzte Turn allein kann teurer sein als die ersten 10 zusammen.

03 — Relevanz

Wann das wirklich zählt

Bei claude.ai als Flatrate-Abo spielt das für den Nutzer keine direkte Rolle. Aber sobald die API direkt genutzt wird – in eigenen Projekten, Skripten, Agenten – schlägt es unmittelbar auf die Kosten durch.

Nutzungsart Token-Kosten relevant? Handlungsbedarf
claude.ai Flatrate Nein (Abo) Keiner
Anthropic API direkt Ja – direkt Hoch
Claude Code / Cursor Ja – pro Session Hoch
Eigene Apps / Agenten Ja – kritisch Sehr hoch
MCP Server Integration Ja – versteckt Hoch

Versteckter Kostentreiber MCP: Jeder registrierte MCP Server fügt seine Tool-Definitionen als Input-Token zu jedem API-Call hinzu – unabhängig davon ob das Tool aufgerufen wird. Bei 5 MCP Servern à 300 Token sind das 1.500 Token overhead pro Turn.

04 — Das zweite Problem

Dokumenten-Drift: Kontext veraltet in Stunden

Neben den Kosten gibt es ein zweites Problem das direkt damit zusammenhängt: Projektdokumentation veraltet schnell – und eine veraltete AI_CONTEXT.md bedeutet dass die KI bei jedem neuen Thread blind neu eingeführt werden muss. 

# Typischer Dokumenten-Drift ohne TRACE:

ARCHITECTURE.md   → veraltet nach 2 Wochen
JOURNAL.md        → veraltet nach 1 Woche
LEARNINGS.md      → veraltet nach 3 Tagen
AI_CONTEXT.md     → veraltet nach Stunden

# Konsequenz: Der Entwickler gibt der KI bei jedem
# neuen Thread dieselbe Basis-Erklärung – und zahlt
# dafür jedes Mal Input-Token.

Das ist doppelt ineffizient: Erst veraltet die Dokumentation, dann muss der Entwickler manuell Kontext aufbauen, dann zahlt er Token für etwas das automatisierbar wäre.

05 — Strategien

Was man dagegen tun kann

Es gibt drei Strategien die sofort helfen – unabhängig von TRACE:

# Strategie 1: Regelmäßige Session-Schnitte
Problem:  Langer Thread = hohe Kosten pro Turn
Lösung:   Neuen Thread starten mit komprimiertem Kontext
Schwelle: ~30.000 Token = Warnung / ~50.000 Token = Reset

# Strategie 2: Token-optimierter Kontext
Problem:  5 separate Docs = hoher Input-Token-Overhead
Lösung:   AI_CONTEXT.md als Single Source of Truth
Ziel:     Maximale Info, minimale Token

# Strategie 3: Prompt Caching (Anthropic API)
Problem:  Wiederkehrender Kontext kostet jedes Mal
Lösung:   Cache-Prefix für statische Kontextblöcke
Ersparnis: Bis zu 90% auf gecachte Input-Token

Claude Code Auto-Compact vs. proaktives Management: Claude Code komprimiert automatisch bei ~80–90% des 200k Token Context Windows – also erst wenn es fast zu spät ist und die Kosten pro Turn bereits sehr hoch sind. Proaktives Session-Management bei 30k–50k Token ist deutlich günstiger.

06 — Die Lösung

TRACE: Token-aware Realtime AI Context Engine

TRACE ist ein MCP Server der lokal auf dem Entwicklerrechner läuft und direkt in Claude Code, Cursor und Codex integriert ist. Es löst beide Probleme gleichzeitig – Kosten-Transparenz und Kontext-Aktualität – ohne selbst API-Kosten zu verursachen.

Kernprinzip: Lokal schwer, API leicht. Alle Berechnungen, Git-Analysen und Dokumentenpflege laufen lokal in Python. Die MCP-Schnittstelle liefert nur komprimierte Ergebnisse zurück.

log_session()
Token-Verbrauch und Kosten pro Session erfassen
get_costs()
Kosten-Report pro Projekt und Zeitraum
check_drift()
Erkennt veraltete AI_CONTEXT.md automatisch
update_context()
Synchronisiert Docs mit Git-History
new_session()
Komprimierten Handoff-Prompt generieren
get_tips()
Aktive Kosten-Optimierungsempfehlungen

Architektur: TRACE läuft als zentraler MCP Server mit einer gemeinsamen ~/.trace/trace.db für alle Projekte. Git Hooks werden über Gits globales Template-System automatisch in jeden neuen Clone installiert – zero manual setup nach der Einrichtung.

Fazit

Token-Kosten in der KI-gestützten Entwicklung sind kein lineares Problem – sie sind ein progressives, das ohne aktives Management exponentiell wächst.

Die meisten Entwickler bemerken das erst wenn die API-Rechnung unerwartet hoch ist. TRACE macht das Problem sichtbar – bevor es teuer wird.

Messe statt schätze. Steuere statt reagiere. Dokumentiere für die KI, nicht nur für Menschen.
