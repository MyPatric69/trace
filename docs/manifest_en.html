01 — The Problem

What most developers don't know

AI-assisted development with Claude, GPT or any other model creates costs that appear simple at first glance: you write a message, the model responds. Input tokens plus output tokens equals cost.

That is wrong. And this misconception costs – literally.

A critical point most developers underestimate: With every new turn in a chat, the entire conversation history is sent to the model as input – not just the new message.

In concrete terms this means:

# What is actually sent to the API on every turn:

Your new message           = input tokens
ALL previous messages      = also input tokens  ← most people miss this
System context + docs      = also input tokens
The model's response       = output tokens

# This happens because LLMs are stateless –
# they have no memory between requests.
# The full context must be re-sent on every single API call.
02 — The Math

Costs don't grow linearly – they grow progressively

A conversation that has accumulated 50,000 tokens after 10 turns costs nearly 100× more at turn 10 than at turn 1 – even though the actual new message is the same length.

Turn New message Context carried forward Total input Cost index
1 100 tokens 0 tokens 100 tokens 1×
5 100 tokens ~4,000 tokens ~4,100 tokens 41×
10 100 tokens ~9,800 tokens ~9,900 tokens 99×
50 100 tokens ~49,800 tokens ~49,900 tokens 499×

The formula is simple: cost(turn N) ≈ cost(turn 1) × N

Real-world example: A Claude Sonnet session with 50 turns and an average of 1,000 tokens per message does not cost 50 × $0.003 = $0.15 – due to accumulated context it costs a multiple of that. The last turn alone can be more expensive than the first 10 combined.

03 — Relevance

When this actually matters

With claude.ai as a flat-rate subscription this has no direct impact on the user. But as soon as the API is used directly – in your own projects, scripts or agents – it hits your costs immediately.

Usage type Token costs relevant? Action needed
claude.ai flat-rate No (subscription) None
Anthropic API direct Yes – immediately High
Claude Code / Cursor Yes – per session High
Custom apps / agents Yes – critical Very high
MCP server integration Yes – hidden High

Hidden cost driver – MCP servers: Every registered MCP server adds its tool definitions as input tokens to every API call – regardless of whether the tool is actually invoked. With 5 MCP servers at ~300 tokens each, that is 1,500 tokens of overhead per turn.

04 — The Second Problem

Documentation drift: context goes stale within hours

Beyond costs there is a second problem directly connected to the first: project documentation goes stale fast – and a stale AI_CONTEXT.md means the AI has to be re-introduced from scratch at the start of every new thread. 

# Typical documentation drift without TRACE:

ARCHITECTURE.md   → stale after 2 weeks
JOURNAL.md        → stale after 1 week
LEARNINGS.md      → stale after 3 days
AI_CONTEXT.md     → stale after hours

# Consequence: the developer gives the AI the same
# baseline explanation at the start of every thread –
# and pays input tokens for it every single time.

This is doubly inefficient: first the documentation drifts, then the developer manually rebuilds context, then pays tokens for something that could be automated.

05 — Strategies

What you can do about it

Three strategies that help immediately – independent of TRACE:

# Strategy 1: Regular session resets
Problem:   Long thread = high cost per turn
Solution:  Start a new thread with compressed context
Threshold: ~30,000 tokens = warn / ~50,000 tokens = reset

# Strategy 2: Token-optimized context
Problem:   5 separate docs = high input token overhead
Solution:  AI_CONTEXT.md as single source of truth
Goal:      Maximum information, minimum tokens

# Strategy 3: Prompt caching (Anthropic API)
Problem:   Recurring context costs tokens every time
Solution:  Cache prefix for static context blocks
Saving:    Up to 90% on cached input tokens

Claude Code Auto-Compact vs. proactive management: Claude Code compresses automatically at ~80–90% of the 200k token context window – by which point the cost per turn is already very high. Proactive session management at 30k–50k tokens is significantly cheaper.

06 — The Solution

TRACE: Token-aware Realtime AI Context Engine

TRACE is an MCP server that runs locally on the developer's machine and integrates directly into Claude Code, Cursor and Codex. It solves both problems simultaneously – cost transparency and context currency – without generating API costs itself.

Core principle: local-heavy, API-light. All calculations, git analysis and documentation maintenance run locally in Python. The MCP interface returns only compressed results.

log_session()
Record token usage and cost per session
get_costs()
Cost report per project and time period
check_drift()
Automatically detects stale AI_CONTEXT.md
update_context()
Synchronises docs with git history
new_session()
Generate compressed handoff prompt
get_tips()
Active cost optimisation recommendations

Architecture: TRACE runs as a central MCP server with a shared ~/.trace/trace.db for all projects. Git hooks are automatically installed into every new clone via Git's global template system – zero manual setup after initial configuration.

Conclusion

Token costs in AI-assisted development are not a linear problem – they are a progressive one that grows exponentially without active management.

Most developers only notice this when the API bill is unexpectedly high. TRACE makes the problem visible – before it becomes expensive.

Measure instead of guess. Control instead of react. Document for the AI, not just for humans.
